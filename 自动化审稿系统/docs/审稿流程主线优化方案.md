# 审稿流程主线优化方案

> **状态：已完成** (2026-05)
> - v5 数据库迁移已执行（`migrate_v5_workflow.py`）：`review_rounds`、`author_notifications` 表已创建，`submissions` 增加 `workflow_stage`/`current_round`/`final_decision` 等字段
> - 前端已重构：`submission_detail.html` 改为轮次卡片主线，`review_round_manage.html` 实现三轮共用管理页
> - 后端 helper `get_round_display_status`/`get_round_next_action` 已实现
> - 编辑提交→主编决定→作者回复→返修闭环已打通

本文档供 coding agent 读取执行。目标是把现有“稿件管理 + 派稿管理”系统，优化为以论文为中心的一审、二审、三审流程工作台。

## 1. 背景与目标

用户是一名法学期刊主编。编辑部通过邮箱收发稿件，传统工作中需要人工完成收稿、匿名化、派稿、接收审稿意见、回复作者、处理返修、继续二审三审等流程。

目标不是建立一套强约束的“合法性规则引擎”，而是建立一套清楚、连续、可人工介入的内部审稿系统：

```text
邮箱收稿
  -> 主编核对稿件
  -> 主编匿名化处理
  -> 一审派给 1 名对应方向编辑
  -> 编辑返回审稿意见、评分表、批注版论文
  -> 主编汇总决定：通过 / 退稿 / 返修
  -> 如退稿：回复作者后流程结束
  -> 如返修：作者返回修改稿，进入二审
  -> 如通过：直接进入二审
  -> 二审派给 2 名编辑
  -> 主编汇总决定：通过 / 退稿 / 返修
  -> 如返修或通过：进入三审
  -> 三审派给 2 名编辑
  -> 主编汇总决定：通过 / 退稿
  -> 如通过三审：审稿流程结束
```

核心要求：

- 以“论文/稿件”为中心，不以零散派稿记录为中心。
- 一篇论文详情页应能看清它处在第几轮、每轮派给谁、谁返回了什么、主编做了什么决定、下一步该做什么。
- 不要硬性阻止主编跨轮派稿或特殊处理，但界面上要给出清晰的“推荐下一步”。
- 所有文件，尤其是原稿、匿名稿、返修稿、评分表、批注版，都应能按论文和轮次追踪。

## 3. 当前系统的主要短板

### 3.1 稿件状态过于粗糙

当前 `submissions.status` 主要是：

```text
待处理 / 派稿中 / 审稿中 / 返修中 / 已录用 / 已退稿 / 作者撤稿
```

这不能清晰表达：

- 待匿名
- 待派一审
- 一审中
- 待一审主编决定
- 待回复作者
- 待作者返修
- 待派二审
- 二审中
- 待二审主编决定
- 待派三审
- 三审中
- 待三审主编决定
- 三审通过，流程结束

### 3.2 多编辑同轮审稿表达不充分

当前 `assignments.round` 可以写“一审/二审/再审/终审/外审”，但稿件列表中 `progress_by_round = {p["round"]: p}` 会把同一轮次多个编辑压成一条。

这会导致二审、三审派给两名编辑时，列表视图无法完整展示。

### 3.3 派稿记录与主编最终决定混在一起

当前 `assignment_update()` 中，如果某个 assignment 被设为“已通过”，系统会把整个稿件更新为“已录用”。这不适合本工作流。

正确逻辑应是：

- 编辑返回的是“编辑意见/编辑建议”。
- 主编做的是“本轮汇总决定”。
- 二审和三审尤其需要主编综合两名编辑意见后再决定。

所以不能用单个编辑的结果直接决定整篇论文最终状态。

### 3.4 编辑端提交后没有自动映射为可汇总结果

当前编辑提交审稿后，assignment 状态变为 `已返回`，但 `result` 字段并不一定同步为通过/返修/退稿。主编还要再去派稿管理里手动更新。

这可以保留人工确认，但详情页应明确提示：

```text
该编辑已返回意见，待主编汇总本轮决定
```

### 3.5 缺少“轮次汇总”概念

一篇论文的每一轮应有一个汇总节点：

```text
一审：编辑 A 返回 -> 主编决定 通过/返修/退稿
二审：编辑 B、C 返回 -> 主编决定 通过/返修/退稿
三审：编辑 D、E 返回 -> 主编决定 通过/退稿
```

当前系统只有 `assignments`，缺少 `review_rounds` 或类似表来记录每一轮的主编决定、回复作者状态、进入下一轮状态。

### 3.6 返修稿版本没有主线记录

返修后作者会返回修改稿。当前邮件收稿可以创建新稿件，但缺少把返修稿关联回原论文的机制。

需要支持：

```text
原稿
匿名稿
一审返修稿
二审返修稿
最终稿
```

这些不应变成完全独立的新投稿，而应作为同一篇论文的不同版本。

### 3.7 主编回复作者未形成明确任务

编辑返回后，主编需要回复作者。当前系统能记录审稿结果，但缺少：

- 待回复作者
- 已回复作者
- 回复时间
- 回复内容/草稿
- 关联附件

### 3.8 匿名化处理没有状态和文件位

用户工作流明确要求主编先匿名化处理再派稿。当前系统没有明确：

- 是否已匿名
- 匿名稿文件
- 原稿与匿名稿的区分

## 4. 总体优化方向

保留现有表和路由，渐进式增强，不重写系统。

建议新增三类核心概念：

1. **稿件流程状态**：写在 `submissions` 上，让一篇论文知道自己目前处于哪个节点。
2. **审稿轮次表**：记录每轮主编汇总决定。
3. **稿件版本/文件表增强**：记录原稿、匿名稿、返修稿、最终稿等。

目标是形成以下页面主线：

```text
稿件详情页
  顶部：当前阶段 + 推荐下一步
  文件区：原稿 / 匿名稿 / 返修稿 / 审稿返回文件
  一审区：派稿编辑、返回状态、编辑意见、主编决定、作者回复
  二审区：两名编辑、返回状态、编辑意见、主编决定、作者回复
  三审区：两名编辑、返回状态、编辑意见、主编决定
  时间线：收稿、匿名化、派稿、返回、决定、回复作者
```

## 5. 数据库优化建议

### 5.1 增强 submissions

在 `submissions` 表新增字段：

```sql
workflow_stage TEXT DEFAULT '待匿名';
current_round TEXT DEFAULT '一审';
final_decision TEXT;
needs_author_reply INTEGER DEFAULT 0;
author_replied_at TEXT;
anonymized INTEGER DEFAULT 0;
anonymous_file_id INTEGER;
parent_submission_id INTEGER;
```

字段解释：

- `workflow_stage`：更细的流程状态。
- `current_round`：一审、二审、三审。
- `final_decision`：最终通过、最终退稿、作者撤稿等。
- `needs_author_reply`：主编是否需要回复作者。
- `author_replied_at`：回复作者时间。
- `anonymized`：是否已经匿名化。
- `anonymous_file_id`：匿名稿文件 id，指向 `submission_files.id`。
- `parent_submission_id`：用于兼容“误把返修稿录成新稿件”的情况，可后续合并。

推荐 `workflow_stage` 值：

```text
待匿名
待派一审
一审中
待一审决定
待一审回复作者
待作者返修
待派二审
二审中
待二审决定
待二审回复作者
待派三审
三审中
待三审决定
已通过三审
已退稿
作者撤稿
```

注意：不需要严格限制状态跳转，只要界面能人工改。

### 5.2 新增 review_rounds 表

新增表：

```sql
CREATE TABLE IF NOT EXISTS review_rounds (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    submission_id INTEGER NOT NULL REFERENCES submissions(id) ON DELETE CASCADE,
    round_name TEXT NOT NULL,
    status TEXT DEFAULT '未开始',
    chief_decision TEXT,
    decision_date TEXT,
    author_reply_status TEXT DEFAULT '无需回复',
    author_replied_at TEXT,
    revision_due_at TEXT,
    notes TEXT,
    created_at TEXT DEFAULT (datetime('now','localtime')),
    updated_at TEXT DEFAULT (datetime('now','localtime'))
);
```

建议 `round_name`：

```text
一审 / 二审 / 三审
```

建议 `status`：

```text
未开始 / 派稿中 / 审稿返回中 / 待主编决定 / 待回复作者 / 待作者返修 / 已完成
```

建议 `chief_decision`：

```text
通过 / 返修 / 退稿
```

三审界面只显示：

```text
通过 / 退稿
```

不要硬性阻止写返修，但默认 UI 不提供三审返修。

### 5.3 增强 assignments

保留现有 `assignments` 表，但新增字段以便连接到轮次：

```sql
review_round_id INTEGER REFERENCES review_rounds(id);
editor_recommendation TEXT;
returned INTEGER DEFAULT 0;
```

说明：

- `review_round_id`：该派稿属于哪个轮次汇总节点。
- `editor_recommendation`：编辑建议，通过/返修/退稿。
- `returned`：是否已经返回。

兼容逻辑：

- 旧字段 `round` 继续保留。
- 旧字段 `result` 继续保留。
- 新逻辑中 `result` 和 `editor_recommendation` 可以同步。

### 5.4 增强 submission_files

当前已有：

```text
submission_files(id, submission_id, filename, file_path, file_type, source, created_at)
```

建议新增：

```sql
round_name TEXT;
version_label TEXT;
uploaded_by_role TEXT;
related_assignment_id INTEGER;
```

`file_type` 建议统一为：

```text
原稿
匿名稿
返修稿
最终稿
著作权协议
查重报告
审稿评分表
批注版
修改说明
其他附件
```

### 5.5 新增 author_notifications 表

用于记录主编是否已经回复作者：

```sql
CREATE TABLE IF NOT EXISTS author_notifications (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    submission_id INTEGER NOT NULL REFERENCES submissions(id) ON DELETE CASCADE,
    review_round_id INTEGER REFERENCES review_rounds(id),
    notification_type TEXT,
    result_label TEXT,
    subject TEXT,
    body TEXT,
    draft_path TEXT,
    status TEXT DEFAULT '草稿',
    sent_at TEXT,
    created_at TEXT DEFAULT (datetime('now','localtime'))
);
```

状态：

```text
草稿 / 已发送 / 无需发送
```

## 6. 后端流程优化

### 6.1 邮件导入后初始化流程

当 `/api/email/import/<staging_id>` 创建新稿件后：

1. 创建 `submissions`。
2. 保存附件到 `submission_files`。
3. 设置：

```text
workflow_stage = 待匿名
current_round = 一审
anonymized = 0
```

4. 自动创建一条 `review_rounds`：

```text
round_name = 一审
status = 未开始
```

### 6.2 匿名化处理

新增主编操作：

```text
标记已匿名
上传匿名稿
```

路由建议：

```text
POST /submissions/<sid>/anonymize
```

行为：

- 上传匿名稿，写入 `submission_files(file_type='匿名稿')`。
- 设置 `submissions.anonymized=1`。
- 设置 `submissions.workflow_stage='待派一审'`。
- 设置 `anonymous_file_id`。

如果主编暂时不上传匿名稿，也允许只点“标记已匿名”。

### 6.3 派稿时自动归入轮次

`assignment_add` 当前只创建 `assignments`。

优化后：

1. 根据表单选择的 `round` 找到或创建 `review_rounds`。
2. assignment 写入 `review_round_id`。
3. 更新 `review_rounds.status='派稿中'`。
4. 更新 `submissions.workflow_stage`：

```text
一审 -> 一审中
二审 -> 二审中
三审 -> 三审中
```

注意：不限制一审只能 1 人、二审三审只能 2 人，但页面可以提示推荐人数。

### 6.4 编辑提交审稿后

`editor_review()` 当前将 assignment 状态设为 `已返回`。

优化后：

1. 将 `review_opinion` 映射为 `editor_recommendation`：

```text
录用 -> 通过
修改 -> 返修
退稿 -> 退稿
```

2. 设置：

```text
assignments.status = 已返回
assignments.returned = 1
assignments.result = editor_recommendation
```

3. 检查同一 `review_round_id` 下的派稿是否都返回。
4. 如果全部返回，设置：

```text
review_rounds.status = 待主编决定
submissions.workflow_stage = 待一审决定 / 待二审决定 / 待三审决定
```

如果未全部返回，保持：

```text
一审中 / 二审中 / 三审中
```

### 6.5 主编汇总本轮决定

新增主编操作：

```text
POST /submissions/<sid>/rounds/<round_id>/decide
```

表单字段：

```text
chief_decision = 通过 / 返修 / 退稿
decision_date
notes
needs_author_reply
```

行为：

如果退稿：

```text
review_rounds.chief_decision = 退稿
review_rounds.status = 待回复作者
submissions.workflow_stage = 待一审回复作者 / 待二审回复作者 / 待三审决定后回复作者
submissions.status = 已退稿
submissions.needs_author_reply = 1
```

如果返修：

```text
review_rounds.chief_decision = 返修
review_rounds.status = 待回复作者
submissions.workflow_stage = 待一审回复作者 / 待二审回复作者
submissions.status = 返修中
submissions.needs_author_reply = 1
```

如果一审通过：

```text
review_rounds.chief_decision = 通过
review_rounds.status = 已完成
submissions.current_round = 二审
submissions.workflow_stage = 待派二审
```

如果二审通过：

```text
review_rounds.chief_decision = 通过
review_rounds.status = 已完成
submissions.current_round = 三审
submissions.workflow_stage = 待派三审
```

如果三审通过：

```text
review_rounds.chief_decision = 通过
review_rounds.status = 已完成
submissions.workflow_stage = 已通过三审
submissions.status = 已录用
submissions.final_decision = 通过三审
```

注意：返修后不是立即进入下一轮，而是先回复作者、等待作者返回修改稿。用户也可以人工提前派下一轮，不要硬性阻止。

### 6.6 回复作者

新增主编操作：

```text
生成回复草稿
标记已回复
```

路由建议：

```text
POST /submissions/<sid>/rounds/<round_id>/author-reply
```

行为：

- 写入 `author_notifications`。
- 设置 `review_rounds.author_reply_status='已发送'`。
- 设置 `author_replied_at`。
- 如果决定是退稿：`workflow_stage='已退稿'`。
- 如果决定是返修：`workflow_stage='待作者返修'`。

### 6.7 作者返回返修稿

新增操作：

```text
上传返修稿 / 关联邮件暂存记录为返修稿
```

路由建议：

```text
POST /submissions/<sid>/revision
```

行为：

- 保存文件到 `submission_files(file_type='返修稿', round_name='一审' 或 '二审')`。
- 如果当前是“一审返修后返回”：

```text
current_round = 二审
workflow_stage = 待派二审
```

- 如果当前是“二审返修后返回”：

```text
current_round = 三审
workflow_stage = 待派三审
```
