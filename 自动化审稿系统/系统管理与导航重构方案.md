# 系统管理与导航重构方案

本文档供 coding agent 读取执行。目标是重构现有审稿系统的左侧导航、系统管理逻辑与前端显示，使系统从“若干功能页面并列”改为“以收稿和一审、二审、三审流程为主线”的内部审稿工作台。

请先阅读：

```text
/Users/zhangruiming/Desktop/法大研究生学报/自动化审稿系统/AGENTS.md
/Users/zhangruiming/Desktop/法大研究生学报/自动化审稿系统/审稿流程主线优化方案.md
```

## 1. 当前问题

当前左侧导航大致包括：

```text
系统总览
稿件管理
派稿管理
邮件收稿
作者管理
编辑管理
我的审稿
```

问题是：

1. **稿件管理、派稿管理、邮件收稿功能重叠。**
   - 邮件收稿负责收稿入口。
   - 稿件管理展示稿件。
   - 派稿管理展示派稿。
   - 但审稿工作实际不是三个并列模块，而是一条连续流程。

2. **派稿管理不应独立成为主线。**
   - 派稿是某一轮审稿中的动作，不应脱离一审、二审、三审单独存在。

3. **稿件管理概念过宽。**
   - 收稿阶段、匿名化阶段、一审阶段、二审阶段、三审阶段都混在“稿件管理”中，会让主编难以判断下一步。

4. **作者管理与编辑管理属于同一类基础资料。**
   - 二者应归入“人员管理”母目录，而不是左侧并列占用一级入口。

5. **主编端与编辑端状态语言不够统一。**
   - 主编需要看到：待处理、待派稿、待审回、待回复、待作者修改、流程结束。
   - 编辑需要看到：待审核、已提交、已完成。

## 3. 业务逻辑主线

### 3.1 收稿管理

收稿管理负责“投稿进入系统”的全部动作。

来源：

```text
投稿邮箱 -> email_staging -> 主编处理 -> submissions
```

收稿管理包含两个核心任务：

1. **确认登记论文与作者信息**
   - 从邮件中识别标题、作者、学科、联系方式、附件。
   - 主编核对并修正。
   - 确认后创建正式稿件记录。

2. **匿名化处理**
   - 主编对稿件进行匿名化。
   - 可以上传匿名稿，也可以先标记“已匿名”。
   - 匿名完成后，稿件进入一审管理的“待处理/待派稿”状态。

收稿管理中的状态建议：

```text
待登记
待匿名
已匿名待一审
已忽略
已登记
```

收稿管理页面应显示：

```text
邮件标题
识别题名
作者/发件人
学科
附件数量
登记状态
匿名状态
操作
```

操作按钮：

```text
核对登记
上传匿名稿
标记已匿名
进入一审
忽略
```

重要逻辑：

- “邮件收稿”不再作为一级独立模块出现，而应纳入“收稿管理”。
- “稿件管理”中原来的新稿件录入功能，应并入收稿管理或作为“手动登记稿件”按钮。
- 收稿管理完成后，稿件自动进入“一审管理”。

### 3.2 一审管理

一审管理显示所有处于一审阶段的稿件。

进入一审的来源：

```text
收稿管理完成登记 + 匿名化 -> 一审管理
```

一审流程：

```text
待处理 / 待派稿
  -> 主编派给 1 名编辑
  -> 主编端显示：待审回
  -> 编辑端显示：待审核
  -> 编辑审回
  -> 主编端显示：待回复
  -> 主编汇总并回复作者
  -> 退稿：流程结束
  -> 返修：显示待修改后回复
  -> 通过：进入二审管理
```

一审管理状态建议：

```text
待处理
待派稿
待审回
待回复
待修改后回复
已退稿
已进入二审
```

主编端状态含义：

- `待处理`：已进入一审，但尚未派稿或尚未明确下一步。
- `待派稿`：应派给一审编辑。
- `待审回`：已派稿，等待编辑返回意见。
- `待回复`：编辑已返回，等待主编回复作者或确认流转。
- `待修改后回复`：主编已要求返修，等待作者返回修改稿。
- `已退稿`：一审退稿结束。
- `已进入二审`：一审通过或返修后已转入二审。

编辑端状态含义：

- `待审核`：该稿件已经分配给编辑，等待编辑提交审稿意见。
- `已提交`：编辑已提交意见，等待主编处理。
- `已完成`：该轮流程主编已处理完毕。

一审管理页面应显示：

```text
题名
学科
匿名状态
一审编辑
派稿时间
编辑返回状态
编辑建议
主编决定
作者回复状态
下一步
```

操作按钮：

```text
派一审
查看稿件
查看编辑意见
汇总一审决定
回复作者
上传返修稿
进入二审
```

### 3.3 二审管理

二审管理显示所有处于二审阶段的稿件。

进入二审的来源：

```text
一审通过 -> 二审管理
一审返修后作者返回修改稿 -> 二审管理
```

二审流程：

```text
待处理 / 待派稿
  -> 主编派给 2 名编辑
  -> 主编端显示：待审回
  -> 两名编辑端分别显示：待审核
  -> 编辑陆续审回
  -> 未全部审回时仍显示：待审回
  -> 两名编辑均审回后，主编端显示：待回复
  -> 主编汇总并回复作者
  -> 退稿：流程结束
  -> 返修：显示待修改后回复
  -> 通过：进入三审管理
```

二审管理状态建议：

```text
待处理
待派稿
待审回
部分审回
待回复
待修改后回复
已退稿
已进入三审
```

二审管理页面应显示：

```text
题名
学科
二审编辑 A 状态
二审编辑 B 状态
审回数量
编辑建议汇总
主编决定
作者回复状态
下一步
```

操作按钮：

```text
派二审
补派编辑
查看全部意见
汇总二审决定
回复作者
上传返修稿
进入三审
```

重要说明：

- 系统可以提示“二审建议 2 名编辑”，但不要硬性阻止派 1 名或 3 名。
- 主编可以人工处理特殊稿件。
- 二审必须区别“部分审回”和“全部审回”。

### 3.4 三审管理

三审管理显示所有处于三审阶段的稿件。

进入三审的来源：

```text
二审通过 -> 三审管理
二审返修后作者返回修改稿 -> 三审管理
```

三审流程：

```text
待处理 / 待派稿
  -> 主编派给 2 名编辑
  -> 主编端显示：待审回
  -> 编辑端显示：待审核
  -> 两名编辑均审回
  -> 主编端显示：待回复 / 待最终确认
  -> 主编汇总决定
  -> 退稿：流程结束
  -> 通过：审稿流程结束
```

三审管理状态建议：

```text
待处理
待派稿
待审回
部分审回
待最终确认
已退稿
已通过
```

三审结果默认只有：

```text
通过
退稿
```

注意：

- UI 默认不显示“返修”。
- 后端不要因特殊情况下出现返修而崩溃，但不要主动引导三审返修。

三审管理页面应显示：

```text
题名
学科
三审编辑 A 状态
三审编辑 B 状态
审回数量
编辑建议汇总
主编最终决定
下一步
```

操作按钮：

```text
派三审
补派编辑
查看全部意见
汇总三审决定
确认通过
确认退稿
```

## 4. 人员管理

作者管理和编辑管理应合并为“人员管理”母目录。

建议前端结构：

```text
人员管理
  Tab 1：作者
  Tab 2：编辑
```

也可以保留两个子路由：

```text
/people/authors
/people/editors
```

短期兼容方案：

- 保留旧路由 `/authors` 和 `/editors`。
- 新增 `/people` 页面。
- 左侧只显示“人员管理”。
- `/people` 页面内部用 tab 嵌入作者列表和编辑列表。

人员管理页面应支持：

作者：

```text
搜索作者
查看作者投稿
编辑作者信息
```

编辑：

```text

查看负责方向

设置密码
停用/启用
```
